# Hi-Voicer 设计规格

日期：2026-05-28

## 目标

Hi-Voicer 是一个面向 Windows 的离线中文语音输入和文件转录软件。它参考 CapsWriter-Offline 的核心思路，但目标是做成一个更友好、启动更快、体验更像普通桌面软件的产品。

用户不应该感受到“服务端”和“客户端”的割裂。用户只需要打开一个软件，让它在托盘后台运行，然后按住快捷键说话，松开后自动把中文输入到当前窗口；也可以把音频或视频文件拖进软件，生成转录文本或字幕。

## 产品原则

1. 只显示一个软件：不弹黑色命令行窗口，不让用户手动启动服务端。
2. 离线优先：模型准备好以后，语音输入和文件转录不依赖网络。
3. 启动体验要快：界面先打开，模型在后台加载和预热。
4. 默认就能用：默认按住 CapsLock 说话，界面清楚显示当前是否可用。
5. 后续好扩展：识别引擎、热词、替换规则、LLM 润色、多语言，都要做成可替换模块。

## 推荐技术架构

采用 **Tauri + Rust + React**。

```text
React / TypeScript 界面
        |
Tauri 命令和事件通信
        |
Rust 应用核心
  - 软件启动和退出
  - 托盘和窗口控制
  - 全局快捷键
  - 麦克风录音
  - 文件转录任务队列
  - 设置保存
  - 热词和替换规则
  - ASR 识别引擎适配层
        |
本地离线 ASR 运行时
  - 优先考虑 sherpa-onnx 或 ONNX Runtime
  - 模型文件放在软件数据目录或便携模型目录
```

内部可以有“后台识别核心”，但用户看不到单独的服务端。如果识别核心崩溃或需要重启，界面只显示“识别核心正在重启”，然后软件内部自动恢复。

## 为什么不优先用 Python 或 Electron

Python 很适合做 AI/语音识别原型，相关库也多。但如果要做成普通用户能安装使用的软件，Python 容易遇到启动慢、打包大、依赖复杂、环境不稳定的问题。

Electron 很适合快速做漂亮界面，因为它本质上是把网页打包成桌面软件。但 Electron 会带上 Chromium 和 Node.js，软件体积、内存占用、启动成本和安全面都会更大。

Tauri 的思路是：界面仍然用前端技术来做，底层系统能力交给 Rust。这样既能做出友好的界面，又能让后台、快捷键、录音、托盘、打包这些能力更轻、更稳。缺点是前期开发难度比 Python 和 Electron 高一些。

## 第一版范围

第一版目标是做出完整可用的软件闭环，不是只做一个界面 Demo。

第一版包含：

1. 首页：显示当前状态、快捷键、模型状态、最近一次识别结果。
2. 托盘模式：可以显示/隐藏窗口、开始/停止监听、打开设置、退出软件。
3. 按住快捷键说话：默认使用 CapsLock。
4. 可选支持鼠标侧键：如果 Windows 输入库支持稳定，再加入。
5. 离线中文语音识别：通过 ASR 引擎适配层实现。
6. 自动输入文字：优先直接输入到当前窗口，不行时用剪贴板粘贴兜底。
7. 文件拖拽转录：支持把音频或视频拖进软件。
8. 导出格式：`.txt`、`.srt`、`.json`。
9. 模型检测：模型缺失或无效时，显示清楚的配置引导。
10. 热词表和基础替换规则。
11. 设置项：快捷键、模型目录、输出目录、输入方式、是否保存录音、是否开机启动。
12. 本地日志和诊断页面。

第一版不做：

1. LLM 润色和改写。
2. 云同步。
3. 多设备客户端。
4. 完整字幕编辑器。
5. 复杂多语言工作流。
6. 浏览器插件联动。

这些功能后续可以加，但第一版先不做，避免软件一开始就变复杂、不稳定。

## 界面结构

界面风格应该是安静、清楚、工具型，不做营销页，也不做花哨的大卡片堆叠。

主要页面：

1. 首页：当前是否可录音、当前快捷键、最近结果、快速操作。
2. 转录：文件拖拽区、任务队列、输出格式、已完成文件。
3. 热词：热词列表、替换规则、导入导出。
4. 设置：快捷键、模型、麦克风、输出方式、开机启动。
5. 诊断：模型状态、麦克风测试、日志、权限检查。

首页必须让用户马上知道三件事：

1. 现在能不能说话？
2. 我要按哪个键？
3. 本地模型准备好了吗？

## 启动流程

软件启动时：

1. 先打开界面。
2. 读取保存的设置。
3. 注册托盘和全局快捷键。
4. 检查麦克风和模型目录。
5. 在后台加载和预热 ASR 模型。
6. 界面显示明确状态：启动中、正在加载模型、可以录音、需要配置模型、麦克风不可用。

重点是：模型加载不能卡住界面。即使模型还没准备好，用户也应该能打开设置和诊断页面。

## 语音输入流程

1. 用户按住快捷键。
2. Rust 录音模块开始采集麦克风音频。
3. 用户松开快捷键。
4. 音频片段交给 ASR 引擎识别。
5. 识别结果经过格式处理、热词替换、规则替换。
6. 最终文字输入到当前正在使用的软件里。
7. 如果用户开启了保存录音，就保存音频和识别结果。
8. 界面更新最近识别记录。

## 文件转录流程

1. 用户把音频或视频拖进“转录”页面。
2. 每个文件生成一个转录任务。
3. 软件在本地提取或解码音频。
4. ASR 按片段识别，支持时间戳时生成字幕段落。
5. 输出 `.txt`、`.srt`、`.json`。
6. 界面显示进度、完成状态和错误信息。

## Rust 核心模块

建议拆成这些模块：

1. `app_state`：全局状态、是否准备好、向界面发送事件。
2. `config`：读取、保存、校验设置。
3. `tray`：托盘菜单和窗口显示隐藏。
4. `shortcuts`：全局键盘和鼠标快捷键。
5. `audio`：麦克风设备和录音。
6. `asr`：识别引擎统一接口和具体实现。
7. `dictation`：按键录音、识别、输入文字的完整流程。
8. `transcription`：文件任务队列、切片、输出文件。
9. `hotwords`：热词和替换规则。
10. `diagnostics`：模型、麦克风、权限、日志检查。

## React 前端模块

建议拆成这些模块：

1. `AppShell`：整体布局、导航、顶部状态。
2. `HomePage`：首页状态和最近识别。
3. `TranscriptionPage`：文件拖拽和任务队列。
4. `HotwordsPage`：热词和规则编辑。
5. `SettingsPage`：设置页。
6. `DiagnosticsPage`：诊断和日志。
7. `components`：按钮、开关、输入框、状态标签、任务行。
8. `lib/api`：封装 Tauri 命令和事件。

## 数据存储

设置保存在本地 JSON 文件里。后续可以支持便携版，比如检测软件旁边是否有 `portable.json` 或 `models` 目录。

建议文件结构：

```text
settings.json
hotwords.json
rules.json
logs/app.log
recordings/YYYY/MM/assets
transcripts
models
```

第一版不需要保存任何密钥。

## ASR 识别策略

第一版先定义一个 ASR 适配接口，再接具体识别引擎。这样后续换模型或换引擎不会影响界面和业务流程。

识别引擎需要支持：

```text
加载模型
识别一段录音
识别一个文件
模型预热
关闭引擎
```

优先考虑 `sherpa-onnx` 或基于 ONNX Runtime 的本地模型，因为它们更适合 Windows 离线分发。具体模型要在早期做一次小验证，比较识别质量、时间戳、GPU 支持、包体大小和许可证。

## 错误处理

软件不能静默失败。常见错误必须在界面上说清楚：

1. 模型缺失或无效。
2. 麦克风不可用。
3. 快捷键注册失败。
4. 当前目标软件拒绝直接输入文字。
5. 文件格式无法解码。
6. 识别核心异常并正在重启。

每个错误都要有一句用户能看懂的提示，同时在诊断页里留下更详细的信息。

## 性能目标

这些是第一版目标，最终要等识别引擎确定后再实测：

1. 普通 Windows 10/11 电脑上，界面 2 秒内出现。
2. 模型还在加载时，设置和诊断页面也能操作。
3. 模型预热完成后，松开快捷键后尽快进入识别流程。
4. 模型加载必须异步，不能让界面卡死。
5. 文件转录在队列中运行，不影响语音输入控制。

## 安全设计

1. 前端界面不能直接访问任意文件系统能力。
2. 所有系统权限都通过明确的 Tauri 命令调用。
3. 正常使用不需要开放本地服务器端口。
4. 模型只从用户配置的本地目录加载。
5. 日志默认不记录完整音频内容，也不记录敏感转录文本。
6. 未来如果加入 LLM 功能，必须明确标注“本地”还是“联网”。

## 测试和验收

第一版至少要验证：

1. 软件能启动，主要页面能切换。
2. 设置能保存，重启后还在。
3. 模型缺失时能正确提示。
4. 快捷键按下和松开的状态变化正确。
5. 麦克风录音能生成有效音频片段。
6. ASR 适配层能用测试音频跑通。
7. 识别结果能经过热词和替换规则。
8. 文件转录能生成 `.txt`、`.srt`、`.json`。
9. 软件能隐藏到托盘并恢复。
10. 前端和 Rust 构建检查通过。

## 实现阶段

### 第一阶段：项目基础

1. 创建 Tauri + React + TypeScript 项目。
2. 做应用外壳、导航和第一版静态界面。
3. 加 Tauri 命令封装和模拟后端数据。
4. 加设置保存。

### 第二阶段：桌面能力

1. 加托盘。
2. 加全局快捷键状态。
3. 加麦克风检测和录音验证。
4. 加诊断页。

### 第三阶段：ASR 集成

1. 定义 ASR 适配接口。
2. 接入第一个本地识别引擎。
3. 加模型检测和预热。
4. 加语音输入完整流程。

### 第四阶段：文件转录

1. 加文件拖拽任务队列。
2. 加音频切片和输出文件。
3. 加进度显示和取消任务。

### 第五阶段：打包和打磨

1. 加安装包和构建配置。
2. 实测启动表现。
3. 打磨界面、空状态和错误状态。
4. 写模型安装和故障排查说明。

## 待验证的技术决策

这些不是需求不清，而是需要在实现早期用小验证确定：

1. 最终使用哪个 ASR 引擎和模型包。
2. Windows 上哪个全局快捷键库最稳，是否可靠支持鼠标侧键。
3. 自动输入文字的最佳方式和兜底顺序。
4. 第一版安装包是否内置小模型，还是引导用户下载模型。
5. GPU 支持第一版就做，还是先作为高级实验选项。

## 已确认方向

用户已确认采用推荐路线：**Tauri + Rust + React**。

这个方向的目标是：第一版做完整可用闭环，后续功能通过模块扩展，不推倒重来。
