# Hi-Voicer 模型说明

## 录音和转写不是同一件事

录音只负责采集麦克风或系统声音。真正把声音变成文字的是 ASR 引擎和 ASR 模型。Hi-Voicer 的语音输入和文件转写会共用同一类 ASR 模型，只是使用场景不同：

- 语音输入：短音频，优先启动速度和实时性。
- 文件转写：长音频，优先稳定性、准确率和可保存结果。

## CapsWriter 的模型处理思路

CapsWriter-Offline 不是“只下载模型就能识别”。它的本地识别链路是：录音模块生成 WAV，ASR 引擎读取音频，ASR 引擎加载模型文件，再输出文字。不同模型有不同适配器和推理后端。

SenseVoiceSmall 也需要引擎。它只是自己具备标点能力，不代表不需要接入引擎。

## 当前模型列表策略

当前正式列表只保留能够一键配置的模型。Qwen3-ASR 1.7B 已移除，因为它需要更大的稳定本地推理包，不能只靠打开 GPU 开关解决。

模型来源统一使用 Hugging Face。Sherpa-ONNX 运行时来自 Sherpa-ONNX 官方 GitHub Release。Qwen3 高效版使用安装包内置的 llama.cpp `b9964` 纯 CPU 运行时，不携带 CUDA 或 Vulkan 组件。

## 当前实现建议

1. 安装包内置约 33.1 MiB 的最小 Sherpa-ONNX CPU 运行时，只保留离线识别和 Qwen 兼容服务所需的两个静态 EXE。
2. Qwen3 高效版只按需下载 Q8 主模型和音频投影器；llama.cpp CPU 运行组件随安装包提供。
3. 大模型使用 `.part` 断点文件，完成后校验固定大小和 SHA-256，再原子写入模型目录。
4. 文件转写时，WAV 可直接送入 Sherpa-ONNX；MP3、M4A、MP4 等格式需要本机 ffmpeg 转成 16kHz 单声道 WAV。
5. 模型固定存放在应用本地数据目录，运行时从当前安装资源目录动态解析，用户不需要选择或修复路径。
6. llama-server 使用动态本地端口、单任务槽和最多 12 个 CPU 线程，空闲 5 分钟后自动退出。
7. GGUF、SenseVoice 和 Qwen3 兼容版读取配置时均以 `engine.json` 所在目录为准，因此模型文件夹可以整体复制给其他用户。
7. Qwen3 高效版失败时，如果旧 Qwen ONNX 已安装，会自动回退兼容版。

## GPU 加速状态

当前正式路线是 Sherpa-ONNX CPU。CUDA 产品入口已移除，原因是 Windows 分发需要额外匹配 CUDA Toolkit、cuDNN、ONNX Runtime CUDA 依赖，安装包体积和用户环境不确定性都偏高。

GPU acceleration currently focuses on DirectML + SenseVoiceSmall-style ONNX models first. Use the Diagnostics DirectML PoC and CPU vs DirectML benchmark before treating it as reliable.

### Qwen3-ASR 0.6B 加速结论

- Qwen3 高效版使用 60 秒分片和常驻 llama-server；兼容版继续使用 20 秒分片。
- Qwen Sherpa WebSocket daemon 会按照并发档位使用 6、3、2 个 CPU 线程；线程档位改变时重启 daemon，避免并发请求继续复用 6 线程配置而造成过度抢占。
- 2026-07-11 使用同一段 20 秒音频和同一套 INT8 权重实测：Sherpa CPU 6 线程实际解码 5.614 秒；DirectML 在排除约 7 秒 Session 加载后，完整解码仍需 50.177 秒。
- 2026-07-11 验证官方 Q8 GGUF：20 秒样本正文与 ONNX 基本逐字一致，常驻 CPU 服务耗时约 3.15 秒；同次 Sherpa ONNX 解码约 5.84 秒。
- Vulkan 全卸载约 14.2 秒，CPU 音频编码加 Vulkan 文本解码约 6.1 秒，均慢于纯 CPU，因此正式高效版不包含 GPU 运行组件。
- Q8 主模型 767.5 MiB，音频投影器 204.5 MiB；llama.cpp CPU 运行组件压缩约 17.4 MiB、安装后约 44.1 MiB。
- llama.cpp 音频支持仍标记为实验能力，因此兼容版会继续保留，真实录音语料验证通过后再考虑隐藏旧入口。
