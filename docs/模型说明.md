# Hi-Voicer 模型说明

## 录音和转写不是同一件事

录音只负责采集麦克风或系统声音。真正把声音变成文字的是 ASR 引擎和 ASR 模型。Hi-Voicer 的语音输入和文件转写会共用同一类 ASR 模型，只是使用场景不同：

- 语音输入：短音频，优先启动速度和实时性。
- 文件转写：长音频，优先稳定性、准确率和可保存结果。

## CapsWriter 的模型处理思路

CapsWriter-Offline 不是“只下载模型就能识别”。它的本地识别链路是：录音模块生成 WAV，ASR 引擎读取音频，ASR 引擎加载模型文件，再输出文字。不同模型有不同适配器和推理后端。

SenseVoiceSmall 也需要引擎。它只是自己具备标点能力，不代表不需要接入引擎。

## 当前模型列表策略

当前正式列表只保留已经能通过 Sherpa-ONNX 自动配置的模型。Qwen3-ASR 1.7B 已移除，因为它需要稳定的一键本地推理包，不能只靠打开 GPU 开关解决。

模型来源统一使用 Hugging Face。Sherpa-ONNX 运行时来自 Sherpa-ONNX 官方 GitHub Release，它是引擎，不是模型源。

## 当前实现建议

1. 自动安装 Sherpa-ONNX CPU 运行时，默认始终走 CPU 稳定路径。
2. 自动下载 Hugging Face 上的模型文件。
3. 自动生成 `sherpa-onnx-offline.exe` 参数并保存配置。
4. 文件转写时，WAV 可直接送入 Sherpa-ONNX；MP3、M4A、MP4 等格式需要本机 ffmpeg 转成 16kHz 单声道 WAV。
5. 启动时自动发现应用数据目录里已安装的模型，避免重复配置。
6. Qwen3-ASR 1.7B 不再作为当前预设入口，除非后续有稳定的一键本地引擎再重新评估。

## GPU 加速状态

当前正式路线是 Sherpa-ONNX CPU。CUDA 产品入口已移除，原因是 Windows 分发需要额外匹配 CUDA Toolkit、cuDNN、ONNX Runtime CUDA 依赖，安装包体积和用户环境不确定性都偏高。

GPU acceleration now focuses on DirectML + SenseVoiceSmall-style ONNX models first. Use the Diagnostics DirectML PoC and CPU vs DirectML benchmark before treating it as reliable.
