# Hi-Voicer 模型说明

## 录音和转录不是同一个东西

录音不需要 AI 模型，它只是把麦克风声音采集成音频。真正把声音变成文字的是 ASR 模型，也就是转录模型。Hi-Voicer 的语音输入和文件转录会共用同一类 ASR 模型，只是使用场景不同：

- 语音输入：短音频，优先启动速度和实时性。
- 文件转录：长音频，优先稳定性、准确率和可保存结果。

## CapsWriter 的模型处理思路

CapsWriter-Offline 不是“只下载模型就能识别”。它的本地识别链路是：

1. 录音模块先生成 16kHz 左右的 WAV 音频。
2. 本地 ASR 引擎读取音频。
3. ASR 引擎加载模型文件。
4. 引擎输出文字，再由程序粘贴上屏或保存文件。

CapsWriter-Offline 的关键点在 `config_server.py` 和 `core/server/engines`：

- `ServerConfig.model_type` 决定当前用 `qwen_asr`、`fun_asr_nano`、`sensevoice` 还是 `paraformer`。
- `ModelPaths` 写死每个模型需要哪些文件，以及这些文件在 `models` 目录下的相对位置。
- `EngineFactory` 根据 `model_type` 创建对应引擎。
- 不同模型有不同适配器：Paraformer 走 Sherpa-ONNX，SenseVoice 走 ONNXRuntime，FunASR-Nano 和 Qwen3-ASR 走 ONNX + GGUF。
- 引擎声明自己的能力：有没有标点、时间戳、热词。缺标点就额外挂 CT-Transformer，缺时间戳就外挂 Qwen ForcedAligner。

所以 SenseVoiceSmall 也需要引擎。它只是自己具备标点能力，不代表“不用接入引擎”。录音和模型也不是一回事：录音只采集声音，转文字一定要 ASR 引擎和 ASR 模型。

CapsWriter-Offline 的模型下载页放在 GitHub Release，解压后手动放到固定 `models` 目录。Hi-Voicer 按你的要求不走国内站点，模型文件统一从 Hugging Face 取，并由软件自动放进本机应用数据目录。

## 当前模型列表策略

当前列表分两类：

1. Sherpa-ONNX 统一运行时
   适合第一版交付。软件会下载 Sherpa-ONNX 运行时，再下载对应 ONNX 模型文件，并自动生成 `engine.json`。

2. 大模型原始权重
   Qwen3-ASR 1.7B 这类官方原始权重不能直接喂给当前桌面软件，需要稳定的 ONNX/GGUF 推理包。现在先保留入口，不假装已经一键可用。

模型来源统一使用 Hugging Face。Sherpa-ONNX 运行时来自 Sherpa-ONNX 官方 GitHub Release，它是引擎，不是模型源。

## 当前实现建议

第一版优先接 Sherpa-ONNX，不继续走 Vosk。原因是这条路线模型形态明确，Windows 可打包，CPU 可跑，也能用一套运行时覆盖 SenseVoiceSmall、Qwen3-ASR 0.6B、FunASR-Nano、Whisper Base、Paraformer、Zipformer。

实现顺序：

1. 自动安装 Sherpa-ONNX CPU 运行时，默认始终走 CPU 稳定路径。
2. 自动下载 Hugging Face 上的模型文件。
3. 自动生成 `sherpa-onnx-offline.exe` 参数并保存配置。
4. 文件转录时，WAV 可直接送入 Sherpa-ONNX；MP3、M4A、MP4 等格式先由后台 ffmpeg 转成 16kHz 单声道 WAV。
5. 跑通录音、识别、上屏完整流程；语音输入默认不保留临时录音和转录文本。
6. 启动时自动发现应用数据目录里已安装的模型，避免换电脑或设置丢失后重复配置。
7. 后续再补 Qwen3-ASR 1.7B 的稳定本地推理包。

## GPU 加速状态

当前开发版已开始接入可选 CUDA 加速，但默认仍是 Sherpa-ONNX Windows x64 CPU 运行时。CUDA 模式会先检测 NVIDIA 环境，再按需下载官方 Windows x64 CUDA runtime；检测、下载、启动或识别失败都会回退 CPU。DirectML 还没有接入，发布说明里只能承诺 CPU 默认稳定和 CUDA 可选试用。
